Events API¶

The history of a Digital Twin is built from events produced by Integrators; these events are declared in a Python package and their base class is AssetEvent …

API reference¶

AssetEvent¶

class ioticlabs.dt.api.event.base.AssetEvent(asset, source=None, time=None, systime=None, offset=-1, version=None, data=None)¶

Base class for asset event types. Covers handling of extra data (schema and validation) and versioning

Only the abstract methods and optionally _validate_data should be implemented/overloaded. The name of the event is derived from the class name itself which must follow camel case and is restriced to ASCII.

asset¶

Asset id (str) to which event applies

data¶

Extra data specific to event type and version, or None if not applicable.

Notes:
  • This is the decoded version - encoding/decoding is automated based on applicable version and schema.
  • An Avro schema defining a single primitive null type cannot be distinguished from no extra data at all,
    i.e. both will decode as None.
classmethod name()¶

Name (str) of event type. Must NOT be overridden.

offset¶

Offset (int) of event in event stream for this asset. A value of -1 indicates the offset is not applicable, i.e. the event instance has been created for publishing.

source¶

Source of event (str), e.g. an integrator. Can be None when instance has been created for publishing.

systime¶

Time of event insertion (datetime). Ignored from publishing perspective. Consumed events have this set to a monotonically increasing timestamp which indicates when the event was stored in the system. It is independent from the time field.

time¶

Time of event (datetime). Can be None when instance has been created for publishing, in which case current UTC time will be used. This field has no monotonicity requirement and should be used to store time which has a specific meaning to the event. (E.g. document creation time.)

version¶

Version (int) of this type of event (defaults to _default_version()). Used to indicate changes in extra data format. Message consumers should explicitly check for event version to ensure they are not blindly consuming newer versions in an incompatible manner.
